# DataPay 商业计划书 (Business Plan)

## 1. 执行摘要 (Executive Summary)
DataPay (核心引擎 `wrap402`) 是一个致力于解决“AI Agent M2M (机器对机器) 经济”中微支付断层的基础设施项目。在 AI 代理越来越自主化的时代，Agent 为了完成复杂任务，需要随时调用外部高质量的数据和API（例如：金融风控黑名单、实时股市数据、专业文献检索等）。然而，目前的互联网API生态基于传统的订阅制或Token化，无法让 AI 做到“按需即用即付”。DataPay 旨在让所有数据提供方**零代码**地将现有 API 或静态数据转化为支持 Web3 微支付标准（基于 HTTP 402 Payment Required）或法币小额代扣的自动售卖节点。

## 2. 市场痛点 (Problem)
- **支付基础设施落后**：AI Agent 无法轻易挂载信用卡或支付宝，传统 SaaS 的先开户、买套餐模式阻碍了 Agent 自动发掘和调用长尾数据。
- **数据持有方变现难**：拥有有价值数据的长尾提供商（中小企业、个人开发者）如果想要开发一套复杂的计费、鉴权API网关，技术门槛极高，往往导致数据闲置。
- **缺乏标准化发现机制**：Agent 无法结构化地发现网络上哪些数据是能够付费获取的。

## 3. 解决方案 (Solution)
DataPay 提供了一套开箱即用的轻量级反向代理+支付网关引擎（`wrap402`）。
- **零代码改造**：只需通过 CLI 定义路由和价格（例如 `wrap402 publish ./data.json --price 1`），即可立即生成带有计费墙的 AI 原生 API。
- **自适应 HTTP 402**：Agent 请求时被 402 HTTP 状态码拦截，并返回标准的支付凭据获取路由，Agent 携带支付令牌后重试即可放行。
- **AI 原生发现 (Auto-discovery)**：自动生成 `/.well-known/x402-assets.json`。
- **动态代理引擎**：除了本地数据，也支持配置上游真实 API 的地址，收到钱后再代理调用。

## 4. 目标市场与受众 (Target Market)
1. **数据提供方 (Data Providers)**：包含但不限于金融数据公司、科研机构、个人爬虫开发者、行业指标数据库。
2. **AI 开发者平台 (AI Dev Platforms)**：需要为其框架下的 Agent 生态提供统一的数据交易市场。

## 5. 商业模式 (Business Model)
- **开源/社区版引流**：开源核心路由引擎 `wrap402`，供开发者免费本地部署并处理加密货币原生微支付。
- **SaaS 代管平台**：为非技术背景的数据提供方提供全托管上线服务，抽取交易手续费 (例如 5%-10%)。
- **企业私有化部署**：针对具有大型机密内部数据孤岛的集团企业，提供内网部署的联邦计费和结算总线。

## 6. 竞争优势 (Competitive Advantage)
- **协议级切入**：相较于做一个“API应用商店”，DataPay 直接从 HTTP 的 402 协议切入，无缝兼容所有遵循该标准的大模型和开发框架。
- **极佳的开箱体验**：配备了极客风格的 Web Dashboard 控制台，实时可视化订单流水、资产状态，并支持内存级热重载，拥有极低的上手门槛。

## 7. 里程碑计划 (Milestones)
- **Phase 1**: 完成 `wrap402` 基础 CLI 及 Web Dashboard 概念验证 (POC)。 *(已完成)*
- **Phase 2**: 开源首个版本，并发布针对 LangChain / AutoGPT 等主流 AI 框架的 Client SDK。
- **Phase 3**: 推出 DataPay SaaS 云端托管服务，支持法币信用卡预授权池。
- **Phase 4**: 促成行业机构联合起草标准化的 “LLM Data Monetization Protocol”。
